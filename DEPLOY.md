# AirVault Concierge — Production Deployment Guide

**Architecture:** PayloadCMS 3 + Next.js 15 backend on **Render** · Next.js 14 frontend on **Vercel**

```
Browser → Vercel (frontend) → rewrites /api/* /ws/* /graphql → Render (airvault-app)
                                                                       ↓
                                                            Render Postgres + Redis
```

---

## Prerequisites

- GitHub repository with this monorepo pushed
- [Render](https://render.com) account (free tier works)
- [Vercel](https://vercel.com) account (free tier works)

---

## Step 1 — Push to GitHub

```bash
git add .
git commit -m "chore: production deployment configs"
git push origin main
```

## Step 2 — Deploy Backend via Render Blueprint

Render reads `render.yaml` at the repo root and automatically provisions:
- **`airvault-postgres`** — managed Postgres (free plan)
- **`airvault-redis`** — managed Redis (free plan)
- **`airvault-app`** — PayloadCMS + Next.js Docker service (starter plan, Oregon region)

### 2.1 Create a new Blueprint

1. Go to [Render Dashboard](https://dashboard.render.com) → **New** → **Blueprint**
2. Connect your GitHub repo
3. Render detects `render.yaml` and shows the three resources — click **Apply**

### 2.2 Fill in manual secrets

After the Blueprint is created, go to **airvault-app → Environment** and set:

| Variable | Value |
|---|---|
| `STRIPE_SECRET_KEY` | `sk_live_...` or `sk_test_...` |
| `RAPIDAPI_KEY` | Your RapidAPI key for flight data |
| `KAFKA_BOOTSTRAP_SERVERS` | Upstash Kafka URL (leave blank to disable) |

> All other env vars (`DATABASE_URI`, `DATABASE_URL`, `REDIS_URL`, `PAYLOAD_SECRET`) are auto-provisioned by Render from the Blueprint.

### 2.3 Note your Render service URL

After the first deploy succeeds, your backend URL will be:
```
https://airvault-app.onrender.com
```
(or a custom domain if you configure one — see Step 4)

---

## Step 3 — Deploy Frontend via Vercel

### 3.1 Import the repo

1. Go to [Vercel Dashboard](https://vercel.com/new) → **Import Git Repository**
2. Select your GitHub repo
3. Set **Root Directory** to `frontend`
4. Framework Preset: **Next.js** (auto-detected)

### 3.2 Add environment variable

In the Vercel project settings (or during import), add:

| Variable | Value |
|---|---|
| `NEXT_PUBLIC_API_URL` | `https://airvault-app.onrender.com` |

> Replace with your actual Render URL if it differs from the default.

### 3.3 Deploy

Click **Deploy**. Vercel builds `frontend/` and publishes to a `.vercel.app` domain.

The `vercel.json` rewrites automatically proxy:
- `/api/:path*` → Render backend
- `/ws/:path*` → Render backend (WebSocket)
- `/graphql` → Render backend

---

## Step 4 — Custom Domains (Optional)

### Render (backend)

1. **airvault-app → Settings → Custom Domains** → Add `api.yourdomain.com`
2. Add a CNAME record at your DNS provider:

| Type | Name | Value |
|---|---|---|
| `CNAME` | `api` | `airvault-app.onrender.com` |

3. Update `NEXT_PUBLIC_SERVER_URL` and `NEXT_PUBLIC_API_URL` env vars on Render to `https://api.yourdomain.com`

### Vercel (frontend)

1. **Project → Settings → Domains** → Add `yourdomain.com`
2. Add a CNAME record at your DNS provider:

| Type | Name | Value |
|---|---|---|
| `CNAME` | `www` | `cname.vercel-dns.com` |
| `A` | `@` | `76.76.21.21` |

3. Update `NEXT_PUBLIC_API_URL` on Vercel to `https://api.yourdomain.com`

## Step 5 — Verification

Replace `RENDER_URL` with your actual Render service URL.

```bash
RENDER_URL=https://airvault-app.onrender.com

# Health check
curl "$RENDER_URL/api/health"

# PayloadCMS admin (should return HTML)
curl -I "$RENDER_URL/admin"

# GraphQL introspection
curl -X POST "$RENDER_URL/api/graphql" \
  -H "Content-Type: application/json" \
  -d '{"query":"{ __typename }"}'

# Flight search
curl "$RENDER_URL/api/v1/flights/search?origin=JFK&destination=LHR&date=2025-06-01"

# Meals
curl "$RENDER_URL/api/v1/meals"

# Ride request (POST)
curl -X POST "$RENDER_URL/api/v1/rides/request" \
  -H "Content-Type: application/json" \
  -d '{"pickup":"JFK Terminal 4","dropoff":"Manhattan Hotel","passengerName":"Test User"}'
```

## Environment Variables Reference

### Render — `airvault-app`

| Variable | Source | Notes |
|---|---|---|
| `PAYLOAD_SECRET` | Auto-generated by Render | PayloadCMS JWT signing secret |
| `DATABASE_URI` | Auto-linked from `airvault-postgres` | Primary Postgres connection string |
| `DATABASE_URL` | Auto-linked from `airvault-postgres` | Legacy alias — both point to same DB |
| `REDIS_URL` | Auto-linked from `airvault-redis` | ioredis connection string |
| `STRIPE_SECRET_KEY` | **Manual** (Render dashboard) | `sk_live_...` or `sk_test_...` |
| `RAPIDAPI_KEY` | **Manual** (Render dashboard) | Flight data API key |
| `KAFKA_BOOTSTRAP_SERVERS` | **Manual**, optional | Upstash Kafka URL; leave blank to disable |
| `NEXT_PUBLIC_SERVER_URL` | Auto-set by Blueprint | e.g. `https://airvault-app.onrender.com` |
| `NEXT_PUBLIC_API_URL` | Auto-set by Blueprint | Same as `NEXT_PUBLIC_SERVER_URL` |

### Vercel — `frontend`

| Variable | Value | Notes |
|---|---|---|
| `NEXT_PUBLIC_API_URL` | `https://airvault-app.onrender.com` | Must match your Render service URL |

> To add via CLI: `vercel env add NEXT_PUBLIC_API_URL production`

## Local Development

For local development use Docker Compose (not covered here — see `docker-compose.yml`).
The `payload-app/` Next.js config and `frontend/next.config.js` both define local rewrites
that proxy API traffic to `localhost:3000` / the env-configured backend URL.
