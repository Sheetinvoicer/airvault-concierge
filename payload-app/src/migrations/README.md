# Migrations

This directory contains PayloadCMS/Drizzle migration files for the SQLite database.

## Overview

The app uses `@payloadcms/db-sqlite` with the `@libsql/client` driver and stores data in `database.db` at the project root.

## Environment Setup

Ensure the following variables are set in `.env` before running any migration or dev commands:

```env
DATABASE_URL=file:./database.db
PAYLOAD_SECRET=<your-secret>
```

## Running Migrations

### Development — schema push (no migration files required)

This is the recommended approach during local development. It pushes the current schema directly to the database:

```sh
npm run db:push
```

### Production — generate and run migration files

1. **Generate a new migration** (after changing a collection schema):
   ```sh
   npm run payload -- migrate:create --name=<migration-name>
   ```
   This creates a new `.ts` file in `src/migrations/`.

2. **Run all pending migrations**:
   ```sh
   npm run migrate
   ```
   This calls `migrate.mjs` which runs `payload migrate --force`.

3. **Check migration status** (list applied/pending):
   ```sh
   npm run payload -- migrate:status
   ```

## Troubleshooting

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| `ENOENT: database.db` | Wrong working directory | Run commands from project root |
| `Migration already run` | Re-running on existing DB | Use `--force` flag (already set in `migrate.mjs`) |
| App crashes on startup | Schema drift between code and DB | Run `npm run db:push` (dev) or `npm run migrate` (prod) |
| `Cannot find module '@payloadcms/db-postgres'` | Old stale import | Removed — `payload.config.ts` now only imports `sqliteAdapter` |

## Verify After Migration

1. Start the app: `npm run dev`
2. Open `/admin` — the Payload dashboard should load without DB errors.
3. Hit an API route, e.g. `GET /api/users` — should return `200`.
4. Check the SQLite tables exist:
   ```sh
   sqlite3 database.db ".tables"
   ```

## Notes

- Migration files are generated by Payload/Drizzle and should be committed to version control.
- `database.db` is local-only and should **not** be committed (it is in `.gitignore`).
- In production (Docker/Render), run `npm run db:push` in the container entrypoint before starting the server.
